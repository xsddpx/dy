# 项目控制文档

## 文档职责

本文件是跨阶段核心事实、流程路由、输入输出合同和全局硬阻断的唯一事实源。具体执行命令、页面操作、prompt 写法、重试方法和记录字段只保存在对应的 `DOCS/PIPELINE/`、`DOCS/WORKFLOWS/` 或 `DOCS/RUNBOOKS/` 文档；其他文档需要这些规则时引用本文件或目标文档，不复制维护。

## 核心目标

本项目通过 `/xdy` 执行 Anna 三图参考日更主链：生成 9:16、720p、5–7 秒的单人竖屏短视频，尝试上传到 Google Drive 的 My Drive 根目录下 `最终视频/`，并按所选路线发布到抖音和快手。`fast` 与 `auto` 仅作为 `/xdy` 的兼容调用别名，不作为流程名称。

正式视频固定使用三张参考图：`@图1` 锚定 Anna，`@图2` 锚定本次衣柜人台商品图，`@图3` 锚定本次固定环境。视频由 Dreamina CLI 生成，平台发布前必须设置 `内容由AI生成`。衣柜入库工作流生成的正式衣柜图统一上传到 My Drive 根目录下 `衣柜图/`。

## 文档结构

- `DOCS/PIPELINE/`：编号严格等于视频主链执行顺序。
- `DOCS/WORKFLOWS/`：不进入视频主链的独立工作流。
- `DOCS/RUNBOOKS/`：任一阶段按需调用、修复后返回原失败点的运行手册。

具体执行只读取当前路线涉及的文档。

## 视频主链

### 默认 `/xdy`

`PIPELINE/01_RUN_INIT.md` 建档 → `PIPELINE/02_CONTENT_AND_PROMPT.md` 选题、素材锁定与 prompt → `PIPELINE/03_VIDEO_GENERATION.md` Dreamina 提交、查询与下载 → `PIPELINE/04_REVIEW_AND_UPLOAD.md` 内容质检、正式成片与 Drive 上传 → `PIPELINE/05_PUBLISH.md` 双平台发布 → `PIPELINE/06_RUN_FINALIZE.md` 记录收尾。

### `/xdysp`

阶段 01 → 02 → 03 → 阶段 04 的免质检分支，直接整理正式成片并尝试上传 Drive → 阶段 06。发布状态固定为 `not_requested`，不进入阶段 05。

### `/xdy` 不发布与确认分支

- 用户明确要求“本次不发布”时：阶段 04 完成内容质检、正式成片和 Drive 上传后，将发布状态记录为 `not_requested`，直接进入阶段 06。
- 用户明确要求“发布前确认”时：阶段 04 完成内容质检、正式成片和 Drive 上传后，将发布状态记录为 `awaiting_confirmation` 并暂停。用户确认发布后进入阶段 05，再进入阶段 06；用户取消发布后改记 `not_requested`，直接进入阶段 06。
- `awaiting_confirmation` 是暂停状态，不等同于用户已经要求不发布。

## 独立工作流与运行手册

- 衣柜图片化入库：`WORKFLOWS/WARDROBE_INGEST.md`。仅在用户调用 `/xyg`、明确要求根据本次服装参考图创建衣柜条目，或明确指定历史衣柜文档做批量图片化迁移时执行；不自动进入视频主链。
- 评论筛选与回复：`WORKFLOWS/COMMENT_TRIAGE_AND_REPLY.md`。仅在用户明确要求查看或回复评论时执行；不修改视频主链。
- 环境修复：`RUNBOOKS/ENVIRONMENT_REPAIR.md`。默认不执行；任一阶段出现环境、网络、登录、权限、依赖或文件读写问题时进入，修复并复测后返回原失败点。

## 跨阶段合同

- 每次 `/xdy` 或 `/xdysp` 正式运行必须先执行阶段 01，创建唯一 `RUN_ID`、`TEMP/RUN_ID/`、运行记录首条 `run/started` 事件和 `OUTPUT/`。所有 prompt、素材锁定、下载、代理图、报告和记录均绑定本次 RUN_ID；正式成片固定为 `OUTPUT/RUN_ID.mp4`。
- 阶段 02 从 `MATERIAL/wardrobe-images/衣柜图-季节-NNN/` 锁定同一条目的商品图和服装描述；季节固定为 `春季`、`夏季`、`秋季` 或 `冬季`，三位编号在每个季节内独立分配。阶段同时锁定固定角色图与本次环境图，直接生成并校验版本化的 `vid-prompt-vN.txt`。
- 阶段 03 必须按人物、衣柜图、环境的顺序使用同一组三图、同一文件哈希和对应 vid prompt 提交 Dreamina；异步任务必须使用返回的 `submit_id` 查询终态并把产物下载到本次运行目录。TNS 重试不得换图、换款、换动作模板或改写服装事实。
- 阶段 04 只处理阶段 03 本次下载的 MP4。`xdy` 通过内容质检后整理正式成片；`xdysp` 明确跳过内容质检并记录 `not_performed`。两条路线只要产生正式成片，都必须尝试上传 Drive。
- 阶段 05 只处理本次 `OUTPUT/RUN_ID.mp4`，并遵守单平台失败不回滚另一平台、已成功平台不重复发布的规则。
- 阶段 06 汇总本次生成、TNS、质检、Drive 和发布状态并执行命名审计；不得从旧目录推断或续用本次状态。
- `MATERIAL/wardrobe-id-migration.json` 只保存历史衣柜编号到新完整编号的审计映射；新运行记录一律使用 `衣柜图-季节-NNN`，不得回写旧运行记录。

## 全局硬阻断

- 生成前必须具备固定角色图、至少一条通过校验的衣柜文件夹和至少一张正式环境图；衣柜图片与描述编号必须一致，三图路径、顺序、SHA-256 和 prompt 中的 `@图1`、`@图2`、`@图3` 角色必须一致。
- 七段式 prompt 必须通过人物、衣柜、环境、视频约束、构图和人物动作校验；穿搭段必须逐字包含已锁定条目的款式提示词。
- Dreamina 明确返回 TNS/安全拦截时最多收敛到 `v5`；仍无产物则停止。其他环境类失败转环境修复运行手册并复测原失败点。
- `xdy` 发布前必须人工对照固定角色代理图与首中尾三帧；仅胸部体量明显偏小时阻断，其余通过。`xdysp` 不执行该质检，由用户检查成片。
- 任一平台点击发布前必须设置 `内容由AI生成`。单个平台受阻时记录失败并继续尝试另一个平台；已成功平台不得因另一平台失败而回滚。
- 正式成片必须尝试上传 Drive。上传失败需记录原因和补传状态，但不阻断 `xdy` 继续发布或其他路线记录收尾。
- 正式衣柜图本地入库后必须尝试上传 Drive。上传失败不回滚已验证的本地条目，但必须汇报失败原因和待补传状态。
- 会话交付遵守 `AGENTS.md`：附有 Google Drive 链接时不再展示或链接图片；媒体仅在用户明确要求且不违反该规则时内嵌。

## 文档归属与变更规则

- 阶段 01 保存 RUN_ID 与启动建档；阶段 02 保存核心卖点、资产选择和 prompt 规则；阶段 03 保存 Dreamina、查询下载和 TNS 重试；阶段 04 保存内容质检、正式成片和 Drive 上传；阶段 05 保存平台发布和去重；阶段 06 保存汇总、记录与审计。
- `WORKFLOWS/WARDROBE_INGEST.md` 保存衣柜人台商品图生成、入库与 Drive 上传；`WORKFLOWS/COMMENT_TRIAGE_AND_REPLY.md` 保存评论流程；`RUNBOOKS/` 保存环境判断、修复入口与已验证案例。
- `SKILLS/xdy/`、`SKILLS/xdysp/` 与 `SKILLS/xyg/` 保存项目入口技能并由 Git 管理；`~/.codex/skills/` 只保留指向项目技能目录的软链接。技能只保存调用语义和项目文档路由，不重复完整 SOP。
- 项目文档、prompt 规则和示例优先使用正向描述，直接写清期望画面、动作和结果；安全、发布、输入合同和硬阻断可保留必要的排除性边界。
- 修改核心事实、核心卖点或提示词规则前，必须先说明拟修改内容和原因；用户授权仅限其明确范围。
- 流程路由变化时同步更新 `README.md` 中的简短 DAG。
