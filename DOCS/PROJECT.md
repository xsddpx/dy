# 项目控制文档

## 文档职责

本文件是跨模块核心事实、流程路由、输入输出合同和全局硬阻断的唯一事实源。具体执行命令、页面操作、prompt 写法、重试方法和记录字段只保存在对应模块；其他文档需要这些规则时引用本文件或目标模块，不复制维护。

## 核心目标

本项目固定执行 `anna auto/fast` 双图参考日更链路，其中 `auto` 即 `fast`；生成 9:16、720p、时长仅为 `5`、`6` 或 `7` 秒的 Anna 单人竖屏短视频，尝试上传到 Google Drive 的 My Drive 根目录，并按所选路线发布到抖音和快手。`@图1` 锚定 Anna，`@图2` 锚定本次固定环境；视频由 Dreamina CLI 生成，全程保持正常速度、不使用慢动作，平台发布前必须设置 `内容由AI生成`。机器可执行配置以 `MATERIAL/xdy-workflow.json` 为唯一事实源，新运行以 `TOOLS/xdy_flow.py` 为唯一写入口。

## 流程路由

- 默认日更：`MAIN_01_RUN_LIFECYCLE.md` 初始化建档 → `MAIN_02_CONTENT_PROMPT.md` 选题、衣柜、环境锁定与 prompt → `MAIN_03_VIDEO_DELIVERY.md` 生成、质检、成片整理与 Drive 上传 → `MAIN_04_PUBLISH.md` 双平台发布 → 主链 01 记录收尾。用户明确要求“本次不发布”时，主链 03 记录 `not_requested` 后直接回主链 01；用户要求“发布前确认”时，主链 03 记录 `awaiting_confirmation` 并等待明确授权，取得授权后才进入主链 04，等待期间明确取消时更新为 `not_requested` 并回主链 01。
- 只生成视频：主链 01 初始化建档 → 主链 02 → 主链 03 的 `xdysp` 分支生成、不质检并上传 Drive → 主链 01 记录收尾；不进入主链 04。
- 环境修复：`AUX_ENV_REPAIR.md`；默认跳过，仅在出现环境问题或用户明确要求时进入，修复后返回原失败步骤复测。
- 评论任务：`AUX_COMMENT_REPLY.md`；仅在用户明确要求查看或回复评论时独立执行。

以上文件均位于 `DOCS/MODULES/`。具体执行只读取当前路线涉及的模块。

## 跨模块合同

- 每次正式运行先由主链 01 创建唯一 `RUN_ID`。所有 prompt、素材锁定、下载、代理图、报告和记录均绑定本次 `TEMP/RUN_ID/`；正式成片固定为 `OUTPUT/RUN_ID.mp4`，不得从旧目录推断本次状态。
- 主链 02 锁定固定角色图与本次环境图并生成通过校验的七段式 prompt；共享固定人物段显式锁定角色图正面、斜侧面和侧面胸部体量、胸廓前后比例及侧向投影，各角度全程稳定一致，`xdy` 与 `xdysp` 均使用该模板。主链 03 必须使用同一组双图、同一顺序和对应 prompt 提交 Dreamina。
- 用户明确指定衣柜编号、正式环境、动作模板或合法时长时优先采用指定值；未指定项读取最近 12 次成功的 schema v2 运行，各维度优先选择最久未使用项，并禁止复用最近 12 次的完全相同三项组合。并列由本次记录的随机 seed 决定。动作默认时长为 `01→7`、`02→6`、`03→5`、`04→6` 秒；用户覆盖也只能是 `5`、`6` 或 `7` 秒。
- `xdy` 在主链 03 完成质检后进入主链 04；`xdysp` 跳过内容质检和发布。两条路线只要产生正式成片，都必须尝试上传 Drive，并由主链 01 记录结果。
- 环境、网络、登录、权限、依赖和文件读写问题统一转环境修复辅助流程；评论查看与回复独立走评论辅助流程，不修改默认生成流程。
- schema v2 事件使用严格枚举和合法迁移，详细证据只进 `logs/`；历史 schema v1 记录只读，不迁移、不重写、不由新工具续跑。

## 全局硬阻断

- 生成前必须通过主链 03 的合同门禁：具备固定角色图和至少一张正式环境图并锁定本次环境；七段式 prompt 通过人物（含胸部体量一致性锚点）、环境、视频约束、构图和人物动作校验；主链 03 的双图输入、顺序与 prompt 引用一致；时长属于 `5`、`6`、`7`。任一项不满足时不得提交 Dreamina。
- Dreamina 明确返回 TNS/安全拦截时最多收敛到 `v5`；仍无产物则停止。其他环境类失败转环境修复辅助流程并复测。
- 默认内容质检的范围固定且仅包含胸部体量：人工先以可读尺寸查看 Anna 角色图顶部身体参考带，再按相近正面、斜侧面或侧面朝向对照首中尾三帧，以肩宽和胸廓为相对尺度判断体量、胸廓前后比例与侧向投影。任一清晰可判断帧明显偏小，或三帧均无法判断时阻断；其余通过。脸部身份、手指、肢体、服装、道具、动作、画面异常和 prompt 还原度均不检查、不评价、不汇报；只有用户另行明确指定其他检查项时才扩展范围。`xdy` 发布前执行该质检；`xdysp` 默认不质检，用户事后要求质检时也遵守同一单项范围。
- 任一平台点击发布前必须设置 `内容由AI生成`。单个平台受阻时记录失败并继续尝试另一个平台；已成功平台不得因另一平台失败而回滚。
- 平台发布只使用 CDP 直传、无位置和中间帧封面，实际应用标签最多 4 个；仅导航超时可自动重试一次，登录、验证码、安全验证、AI 声明和未知错误不得自动重试。
- 正式成片必须尝试上传 Drive。上传失败需记录原因和补传状态，但不阻断 `xdy` 继续发布或 `xdysp` 记录收尾。
- `awaiting_confirmation` 只可等待，不能收尾。会话交付一旦附上 Google Drive 链接，同一交付中不再展示、附上或链接任何图片，包括固定角色代理图和首中尾帧。

## 文档归属与变更规则

- 主链 01 保存 RUN_ID、记录与摘要；主链 02 保存核心卖点、资产选择和 prompt 规则；主链 03 保存 Dreamina、质检、成片整理与 Drive 上传；主链 04 保存平台发布和去重。环境修复与评论分别保存在对应辅助流程文档中。
- `SKILLS/xdy/` 与 `SKILLS/xdysp/` 保存项目入口技能并由 Git 管理；`~/.codex/skills/` 只保留指向项目技能目录的软链接。技能只保存调用语义和项目文档路由，不承载本仓库执行细则。
- 项目文档、prompt 规则和示例优先使用正向描述，直接写清期望画面、动作和结果；安全、发布、输入合同和硬阻断可保留必要的排除性边界。
- 修改核心事实、核心卖点或提示词规则前，必须先说明拟修改内容和原因；用户授权仅限其明确范围。
- 流程路由变化时同步更新 `README.md` 中的简短 DAG。
