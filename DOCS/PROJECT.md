# 项目控制文档

## 文档职责

本文件是跨模块核心事实、流程路由、输入输出合同和全局硬阻断的唯一事实源。具体执行命令、页面操作、prompt 写法、重试方法和记录字段只保存在对应模块；其他文档需要这些规则时引用本文件或目标模块，不复制维护。

## 核心目标

本项目固定执行 `anna auto/fast` 三图参考日更链路，其中 `auto` 即 `fast`；生成 9:16、720p、5–7 秒的 Anna 单人竖屏短视频，尝试上传到 Google Drive 的 My Drive 根目录，并按所选路线发布到抖音和快手。`@图1` 锚定 Anna，`@图2` 锚定本次衣柜人台商品图，`@图3` 锚定本次固定环境；视频由 Dreamina CLI 生成，平台发布前必须设置 `内容由AI生成`。

## 流程路由

- 默认日更：`MODULE_01_REFERENCE.md` 选题、衣柜与 prompt → `MODULE_02_DREAMINA_VIDEO.md` 生成、质检与 Drive 上传 → `MODULE_03_PUBLISH.md` 双平台发布 → `MODULE_04_RECORDS.md` 记录收尾。
- 只生成视频：模块 01 → 模块 02 的 `xdysp` 分支生成、不质检并上传 Drive → 模块 04；不进入模块 03。
- 衣柜图片化入库：`MODULE_05_WARDROBE_IMAGE_ASSETS.md`；仅在用户调用 `/xyg` 或明确要求根据本次服装参考图创建衣柜条目时执行，不进入模块 01–04。
- 环境修复：`MODULE_00_ENV_REPAIR.md`；默认跳过，仅在出现环境问题或用户明确要求时执行。
- 评论任务：`MODULE_06_COMMENT_REPLY.md`；仅在用户明确要求查看或回复评论时执行。

以上文件均位于 `DOCS/MODULES/`。具体执行只读取当前路线涉及的模块。

## 跨模块合同

- 每次正式运行先由模块 04 创建唯一 `RUN_ID`。所有 prompt、素材锁定、下载、代理图、报告和记录均绑定本次 `TEMP/RUN_ID/`；正式成片固定为 `OUTPUT/RUN_ID.mp4`，不得从旧目录推断本次状态。
- 模块 01 从 `MATERIAL/wardrobe-images/衣柜图-NNN/` 锁定同一条目的商品图和服装描述，同时锁定固定角色图与本次环境图，并生成通过校验的七段式 prompt。
- 模块 02 必须按人物、衣柜图、环境的顺序使用同一组三图、同一文件哈希和对应 vid prompt 提交 Dreamina；TNS 重试不得换图、换款或改写服装事实。
- `xdy` 在模块 02 完成质检后进入模块 03；`xdysp` 跳过内容质检和发布。两条路线只要产生正式成片，都必须尝试上传 Drive，并由模块 04 记录结果。
- 环境、网络、登录、权限、依赖和文件读写问题统一转模块 00；评论查看与回复独立走模块 06，不修改默认生成流程。

## 全局硬阻断

- 生成前必须具备固定角色图、至少一条通过校验的衣柜文件夹和至少一张正式环境图；衣柜图片与描述编号必须一致，三图路径、顺序、SHA-256 和 prompt 中的 `@图1`/`@图2`/`@图3` 角色必须一致。
- 七段式 prompt 必须通过人物、衣柜、环境、视频约束、构图和人物动作校验；穿搭段必须逐字包含已锁定条目的款式提示词。
- Dreamina 明确返回 TNS/安全拦截时最多收敛到 `v5`；仍无产物则停止。其他环境类失败转模块 00 修复并复测。
- `xdy` 发布前必须人工对照固定角色代理图与首中尾三帧；仅胸部体量明显偏小时阻断，其余通过。`xdysp` 不执行该质检，由用户检查成片。
- 任一平台点击发布前必须设置 `内容由AI生成`。单个平台受阻时记录失败并继续尝试另一个平台；已成功平台不得因另一平台失败而回滚。
- 正式成片必须尝试上传 Drive。上传失败需记录原因和补传状态，但不阻断 `xdy` 继续发布或 `xdysp` 记录收尾。

## 文档归属与变更规则

- 模块 00 保存环境与修复经验；模块 01 保存核心卖点、资产选择和 prompt 规则；模块 02 保存 Dreamina、质检与 Drive 上传；模块 03 保存平台发布和去重；模块 04 保存 RUN_ID、记录与审计；模块 05 保存衣柜人台商品图生成与入库规则；模块 06 保存评论流程。
- `SKILLS/xdy/`、`SKILLS/xdysp/` 与 `SKILLS/xyg/` 保存项目入口技能并由 Git 管理；`~/.codex/skills/` 只保留指向项目技能目录的软链接。技能只保存调用语义和项目文档路由，不重复完整 SOP。
- 项目文档、prompt 规则和示例优先使用正向描述，直接写清期望画面、动作和结果；安全、发布、输入合同和硬阻断可保留必要的排除性边界。
- 修改核心事实、核心卖点或提示词规则前，必须先说明拟修改内容和原因；用户授权仅限其明确范围。
- 流程路由变化时同步更新 `README.md` 中的简短 DAG。
